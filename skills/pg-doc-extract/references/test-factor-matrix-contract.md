# 测试因子矩阵契约

## 目标

测试因子矩阵是官方文档到测试点之间的可审阅中间层，回答三个问题：

```text
官方章节覆盖了哪些行为维度？
每个维度有哪些有效值、边界值和异常值？
应该如何组合这些因子生成测试点？
```

先生成因子矩阵，再生成测试点概览。测试点是因子值按组合策略生成的结果；除非用户明确要求快速草稿，不要绕过因子矩阵直接堆测试点。

## 核心建模原则

测试因子必须来自官方文档或用户补充资料描述的“被测空间”，而不是从已有测试点名称反向包装。

正确顺序是：

```text
官方章节原文
  -> 功能空间/语义空间
  -> 测试因子与因子值
  -> 组合策略
  -> 计划测试点
```

不要使用以下反向流程：

```text
旧测试点列表
  -> F01 测试行为目标
  -> 每个值对应一个测试点
```

`F01 测试行为目标`、`本章节 N 个单一测试点`、`已有测试点计划` 这类结果型描述只能作为历史迁移备注或覆盖汇总，不能作为核心因子。

## 路径规则

默认放在：

```text
docs/test-factors/<module>/<official-section-dir>/factor-matrix.md
```

示例：

```text
docs/test-factors/chapter-13-concurrency-control/13.2-transaction-isolation/13.2.1-read-committed/factor-matrix.md
```

## 因子类型

优先从官方原文抽取以下类型：

- `object`：对象、语句或被测实体，例如 table、transaction、SELECT、UPDATE、savepoint。
- `parameter`：语法选项、配置项、隔离级别、锁模式、事务模式。
- `state`：系统或事务状态，例如 active、waiting、aborted、committed、idle in transaction。
- `condition`：前置条件或触发条件，例如并发提交前、提交后、冲突存在。
- `behavior`：行为或语义，例如可见性、锁等待、错误处理。
- `boundary`：边界值、极限、空值、最小/最大值、回卷点、超时点。
- `exception`：错误、限制、不支持或版本差异。
- `diagnostic`：可观察视图、catalog、状态确认方式。
- `no-test`：官方章节有描述但不适合自动化测试或不需要测试。

## 因子抽取顺序

按以下顺序建模，缺失的维度可以省略，但不要跳过真实存在的对象、参数、状态或边界：

1. 对象/语句：官方章节涉及哪些对象、命令、系统视图或实体。
2. 参数/选项：有哪些模式、选项、隔离级别、锁模式、配置项或语法分支。
3. 状态/条件：需要什么事务状态、提交状态、并发关系、可见性条件或环境条件。
4. 行为语义：官方承诺或限制的核心行为是什么。
5. 边界值：最小/最大值、首次/再次、提交前/提交后、超时、回卷、空值等。
6. 异常/错误：禁止场景、冲突、错误码、权限或不支持行为。
7. 诊断观测：如何从系统视图、函数、catalog 或可见结果确认行为。
8. no-test：官方说明但不适合自动化或已经重复覆盖的内容。

SQL 语法类章节应优先按语法结构和对象空间建模。例如 `CREATE TABLE` 可抽取：

```text
表类型、创建方式、列定义、约束、继承/分区、存储参数、权限/所有者、异常边界
```

并发/事务语义类章节应优先按语义空间建模。例如 transaction isolation 可抽取：

```text
隔离级别、事务访问模式、并发关系、语句类型、可见性现象、冲突结果、非事务对象、诊断观测
```

## 优先级

每个因子和因子值都要标注优先级：

```text
P0 core       官方核心语义或高风险行为，必须覆盖。
P1 important 重要分支、常见组合或明确边界，应该覆盖。
P2 extended  扩展组合、低频路径或补充诊断，按成本选择。
P3 no-test   不自动化或暂不测试，必须说明原因。
```

优先级理由必须来自官方文档含义、用户风险、边界风险或重复覆盖判断，不要只写“重要”。

```markdown
# <chapter-number> <chapter-title> 测试因子矩阵

- source_type: official-doc / local-knowledge / user-supplement / legacy-derived / needs-confirmation
- source_ref: <官方文档章节、知识库文件、用户补充说明或历史目录>
- source_note: <说明该来源是否为官方章节；扩展测试点不得写成官方章节>

## 官方章节结构
- <chapter-number> <chapter-title>
  - <subsection-number> <subsection-title>

## 官方范围摘要
<说明本矩阵覆盖的官方段落范围。>

## 因子清单
| 因子ID | 因子类型 | 因子名称 | 有效值/状态/边界 | 优先级 | 官方/来源依据摘要 | 备注 |
|---|---|---|---|---|---|---|
| F01 | behavior | <name> | <values> | P0 | <summary> | <note> |

## 因子值细化
### F01 <因子名称>
| 值ID | 值 | 类型 | 优先级 | 依据 | 边界/异常说明 |
|---|---|---|---|---|---|
| F01-V01 | <value> | normal/boundary/exception | P0 | <summary> | <note> |

## 组合策略
| 策略ID | 组合方法 | 适用因子 | 目的 | 生成规则 |
|---|---|---|---|---|
| C01 | single-factor | F01 | 覆盖核心行为 | 每个 P0 行为至少 1 个测试点 |
| C02 | pairwise | F01,F02,F03 | 控制组合爆炸 | P0/P1 值做 pairwise，P2 只在必要时加入 |
| C03 | boundary-directed | F04,F05 | 覆盖边界 | 每个边界值单独成点或与最小必要条件组合 |

## 计划测试点
| 测试点ID | 来源组合 | 优先级 | 覆盖因子值 | 生成原因 | 输出 test-point 文件 |
|---|---|---|---|---|---|
| TP01 | C01 | P0 | F01-V01,F02-V01 | <reason> | `<slug>.md` |

## 不生成测试点的因子或组合
| 因子/组合 | 原因 | 后续动作 |
|---|---|---|
| <id> | duplicate-covered/no-test/not-automatable/out-of-scope | <action> |

## 覆盖检查
- 每个 P0 因子值是否至少被一个测试点覆盖。
- 每个 P1 边界或异常是否被覆盖或有明确不测理由。
- P2 因子是否因为成本、重复或环境限制被延后。
- 是否存在组合爆炸；若存在，是否采用 pairwise 或定向组合说明取舍。
```

## 组合方法选择

- `single-factor`：用于官方明确的单一核心行为，避免一个用例测多个主场景。
- `equivalence-class`：用于多个等价值，只选代表值并记录未选值。
- `boundary-directed`：用于边界、异常、最小值、最大值、回卷、空值、提交前后状态。
- `pairwise`：用于多个 P0/P1 因子都有多个值，且全组合成本过高。
- `state-transition`：用于事务状态、锁状态、提交/回滚前后状态变化。
- `risk-based`：用于 P2 扩展场景，只选择高风险或官方特别强调的组合。

## 质量规则

- 因子必须来自官方章节或用户补充资料，不能凭空扩展。
- 因子名称必须是可组合维度，不能是最终测试点名称。
- 不要把旧测试点清单转换成 `F01 测试行为目标` 作为核心因子；历史测试点必须重新映射到真实因子值组合。
- 因子清单中的 `优先级` 如包含多个级别，应写为 `mixed，详见因子值细化`；最终测试点优先级必须来自其覆盖的具体因子值和风险判断。
- 来源必须真实准确：本地知识库、用户补充、历史迁移或扩展章节不能标注为官方文档。
- PostgreSQL 官方文档不存在的章节号，例如用户自建的 `13.8` 扩展目录，必须使用 `source_type: local-knowledge` 或 `user-supplement`，并在标题或摘要中标明“扩展测试点”。
- 如果来源无法确认，使用 `source_type: needs-confirmation`，不要补造官方依据。
- 每个测试点都必须回链到因子值和组合策略。
- 不要用全组合制造大量低价值测试点；优先 P0/P1，P2 写明取舍。
- 边界值、异常值和官方限制不能被主路径吞掉。
- 无测试点章节也要有矩阵，说明哪些内容不测以及为什么。
- 不写 SQL、表结构、会话同步点或具体执行步骤。
